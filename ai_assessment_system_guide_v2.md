# AI 시대 교실 실천 평가 시스템 개발 가이드 v2

## 이 문서의 목적

이 문서는 고등학교 AI/정보 수업에서 사용하는 **7개 실천 활동**의 학생 산출물을 **AI가 평가·분석하는 시스템**을 만들기 위한 가이드입니다. 새 채팅에서 이 문서를 첨부하고 개발을 시작하세요.

각 실천별로 **탁월/우수/보통/미달 수준의 실제 학생 작성 예시**가 포함되어 있어, AI가 평가 기준을 정확히 학습할 수 있습니다.

## Part 1: 배경 — 교육 철학과 구조

### 핵심 교육 철학

1. **"AI는 제안 OK, 결정은 X"** — AI가 답을 줄 수 있지만, 최종 판단과 선택은 학생이 한다
2. **"같이 배운다" = 같은 어려움을 겪는 것** — 공유된 경험이 관계의 재료
3. **"자기 이해에서 시작"** — 교육 목적은 자아발견과 자아실현
4. **결과물이 아니라 과정을 본다** — AI가 만든 결과물은 학생 역량의 증거가 아님

### 평가 프레임워크 (FACT 기반)

| 요소 | AI 사용 | 평가 내용 | 해당 실천 |
|------|---------|----------|----------|
| F (Fundamental) | AI 없이 | 기초 역량, 핵심 개념 이해 | 실천5(구술면접) |
| A (Applied) | AI 활용 | 실제 문제 해결, 인간-AI 협업 과정 | 실천4(AI 활용 일지) |
| C (Conceptual) | AI 없이 | 개념적 이해 독립 검증 | 실천5(구술면접) |
| T (Thinking) | AI 활용 | AI 산출물 비판적 평가, 결정력 | 실천2,3,4,7 전반 |

### 실천 1: 불편함 수집

**산출물**: 활동지 (불편함 목록 + 분류 + 최종 선택 + 이유)

#### 루브릭

| 항목 | 탁월 (4) | 우수 (3) | 보통 (2) | 미달 (1) |
|------|---------|---------|---------|---------|
| 불편함의 구체성 | 시간·장소·빈도·영향이 명확 | 상황은 구체적이나 수치 부족 | 존재하지만 모호 | 한 단어 수준 |
| 분류의 적절성 | 분류 이유를 설명 가능, 다면적 인식 | 대체로 적절한 분류 | 분류했으나 근거 불명확 | 분류 시도 안 함 |
| 선택 이유의 깊이 | 개인 경험+영향 범위+해결 가능성 고려 | 이유가 있으나 한 측면만 | "제일 불편해서" 수준 | 미작성 |

#### 수준별 예시

**탁월 (4점)**
```
불편함: 등교 시간(7:50-8:00) 정문 앞 횡단보도에서 신호가 90초인데, 한 번에 건너지 
못하면 다음 신호까지 90초를 더 기다려야 해서 매일 약 40명의 학생이 지각 위기를 겪는다.
분류: 시간 효율 (+ 고통 감소: 겨울에 추위 속에서 기다리는 것)
선택 이유: 매일 겪는 문제이고, 나만의 문제가 아니라 전교생이 해당됨. 신호 체계 
데이터를 분석하면 해결 방안을 제시할 수 있을 것 같음. 센서로 학생 밀집도를 
측정하면 시청에 민원 근거 자료를 만들 수도 있다고 생각함.
```
→ 평가 근거: 수치(90초, 40명) 포함, 다면적 분류(시간+고통), 선택 이유에 영향 범위(전교생)+해결 가능성(센서 측정)+활용 방안(민원 근거)까지 서술

**우수 (3점)**
```
불편함: 아침에 학교 정문 앞 횡단보도 신호가 길어서 지각할 뻔한 적이 많다.
분류: 시간 효율
선택 이유: 거의 매일 겪는 문제라서. 해결하면 많은 학생이 편해질 것 같다.
```
→ 평가 근거: 상황은 구체적이나 수치 없음(몇 초? 몇 명?), 분류는 적절, 이유에 영향 범위는 있으나 해결 방안 없음

**보통 (2점)**
```
불편함: 횡단보도 신호가 너무 길다
분류: 시간 효율
선택 이유: 제일 불편해서
```
→ 평가 근거: 불편함이 존재하지만 맥락(어디? 언제?) 없음, 이유가 단편적

**미달 (1점)**
```
불편함: 학교
분류: (비어있음)
선택 이유: (비어있음)
```
→ 평가 근거: 불편함이 구체화되지 않음, 분류·이유 미작성

### 실천 2: AI와 비교하기

**산출물**: 비교표 (AI 결과 vs 내 결과) + 최종 선택 + 이유

#### 루브릭

| 항목 | 탁월 (4) | 우수 (3) | 보통 (2) | 미달 (1) |
|------|---------|---------|---------|---------|
| "나만 찾은 것"의 질 | 현장 맥락 기반의 고유한 발견 2개+ | 고유한 발견 1개 | AI와 비슷하지만 약간 다른 것 | 비어있음 |
| 비교 분석 깊이 | AI와 자기 발견의 차이를 구조적으로 분석 | 차이를 인식하지만 분석 부족 | 단순 나열 | 미작성 |
| 선택 이유 (결정력) | 가치관·경험 기반 논리적 이유 | 이유가 있으나 깊이 부족 | "이게 좋아서" | 미작성 |

#### 수준별 예시

**탁월 (4점)**
```
[나만 찾은 것]
1. 3층 남자화장실 3번 칸 문이 안 잠김 — AI는 "화장실 청결" 같은 일반적 문제만 
   제시했는데, 구체적으로 어떤 칸이 고장인지는 매일 쓰는 사람만 안다.
2. 점심시간에 매점이 12:05에 여는데 4교시가 12:10에 끝나서, 12:10~12:13 사이 
   3분이 승부인데 AI는 이 타이밍을 모른다.

[비교 분석]
AI는 "통학 시간", "학업 스트레스" 같은 누구나 알 수 있는 일반적 불편함을 잘 정리했다.
하지만 우리 학교만의 구체적 상황(특정 시설 고장, 시간표와 급식 시간의 충돌)은 
전혀 찾지 못했다. AI의 답변은 "어느 학교에나 해당되는 것"이고, 
내가 찾은 것은 "서울고에서만 해당되는 것"이다.

[최종 선택] 매점 3분 타이밍 문제
[이유] AI가 제안한 "급식 대기 시간"보다 더 구체적이고, 실제로 데이터를 모을 수 있다.
매점 앞에 센서를 달면 시간대별 혼잡도를 측정할 수 있고, 이 데이터로 매점 오픈 
시간 조정을 학교에 건의할 수 있다. 내가 매일 겪는 문제이기도 하다.
```

**우수 (3점)**
```
[나만 찾은 것]
1. 점심시간에 매점 줄이 너무 길다 — AI도 비슷한 걸 말했지만 구체적 시간은 나만 안다.

[비교 분석]
AI가 찾은 것과 내가 찾은 것이 절반 정도 겹쳤다. AI가 못 찾은 건 
우리 학교의 구체적 상황이다.

[최종 선택] 매점 줄 문제
[이유] 매일 겪는 문제이고, 해결하면 좋을 것 같아서.
```

**보통 (2점)**
```
[나만 찾은 것]
딱히 없음. AI랑 비슷했다.

[비교 분석]
AI가 더 많이 찾았다.

[최종 선택] AI가 제안한 "수면 부족"
[이유] 이게 제일 공감돼서.
```

**미달 (1점)**
```
[나만 찾은 것] (비어있음)
[비교 분석] (비어있음)
[최종 선택] (비어있음)
[이유] (비어있음)
```

### 실천 3: 문제 정의서

**산출물**: 7개 항목 문제 정의서

#### 루브릭

| 항목 | 탁월 (4) | 우수 (3) | 보통 (2) | 미달 (1) |
|------|---------|---------|---------|---------|
| 1. 문제 정의 | 한 문장, 측정 가능한 요소 포함 | 명확하지만 측정 요소 부족 | 모호하지만 방향 있음 | 매우 추상적 |
| 2. 구체성 | 시간·장소·빈도·수치 모두 포함 | 일부 수치 포함 | 서술적이나 수치 없음 | 한 줄 미만 |
| 3. 대상 인식 | 영향 범위와 특성 구체적 | 대상 명시하나 구체성 부족 | "학생들" 수준 | 미작성 |
| 4. 긴급성 | 지속 시 구체적 결과를 논리적으로 | 결과 언급하나 논리 부족 | "불편하니까" 수준 | 미작성 |
| 5. AI 활용 | 답변을 구체적으로 요약+맥락화 | 요약했으나 맥락 부족 | 단순 복붙 | 미사용 |
| 6. 비판적 사고 | 구체적 맥락 2개+ 빠진 것 + 이유 | 1개 빈틈 지적 + 설명 | "좀 다른 것 같다" | "없음" |
| 7. 결정 이유 | 가치관/경험 기반 논리적 이유 | 이유 있으나 깊이 부족 | "그냥 이게 좋아서" | 미작성 |

#### 전체 예시: 탁월 (4점)

```
1. 문제는 무엇인가?
4교시 종료 시각(12:10)과 급식 시작 시각(12:00)의 10분 차이로 인해, 
12:10에 급식실에 도착하는 학생은 평균 15분을 대기해야 하며, 
실제 식사 시간이 10분 이하로 줄어드는 문제.

2. 구체적으로 설명하면?
매일 12:10에 급식실에 도착하면 줄이 약 80-100명. 2개 배식구에서 
분당 약 6명 처리하므로 15분 대기. 급식 시간이 12:40까지이므로 
실제 식사 시간은 10-15분. 특히 월·수요일에 인기 메뉴가 나오면 
줄이 더 길어져 대기 시간 20분 이상.

3. 누구에게 문제인가?
4교시가 12:10에 끝나는 1학년 8개 반(약 240명)이 직접 영향을 받음. 
특히 4층에서 내려오는 5-8반은 이동 시간까지 합치면 12:13에 도착해서 
가장 피해가 큼.

4. 왜 지금 풀어야 하는가?
매일 240명이 겪는 반복적 문제. 짧은 식사 시간이 소화 불량, 
오후 수업 집중력 저하로 이어짐. 2학기에 시간표가 바뀌기 전에 
데이터를 모아서 학교에 건의하면 실제로 바뀔 가능성이 있음.

5. AI는 이 문제에 대해 뭐라고 하나?
AI에게 "학교 급식 대기 시간 문제의 원인과 해결책"을 물었음.
원인: 배식 속도 한계, 시간대 집중, 메뉴 다양성 부족으로 분산 실패
해결 제안: (1) 학년별 시차 급식, (2) 배식구 추가, (3) 모바일 사전 주문, 
(4) 메뉴 다양화로 시간 분산

6. AI 답변에서 빠진 것 또는 틀린 것은?
- AI가 제안한 "모바일 사전 주문"은 우리 학교에서 수업 중 휴대폰 사용 금지 
  규정이 있어서 불가능. AI는 이 규정을 모름.
- "메뉴 다양화로 시간 분산"은 우리 학교 급식이 단일 메뉴 체계라 
  구조적으로 불가능. 2개 메뉴를 운영하려면 조리실 증설이 필요한데 
  현실적이지 않음.
- AI는 "4교시가 12:10에 끝난다"는 우리 학교 시간표의 특수성을 모름. 
  대부분의 학교는 12:00에 4교시가 끝남.

7. 내가 이 문제를 고른 이유는?
AI의 해결책은 학교 제도 변경이 필요한 것들인데, 내가 할 수 있는 것은 
"데이터로 증명하는 것"이다. 급식실 앞에 센서를 설치해서 시간대별 
대기 인원을 자동으로 측정하고, 일주일간 데이터를 모아서 시각화하면, 
학교에 시차 급식을 건의할 때 설득력 있는 근거가 된다. 
내가 Pico와 센서를 배우고 있으니 직접 만들 수 있고, 
내가 매일 겪는 문제라 동기부여도 확실하다.
```

#### 전체 예시: 우수 (3점)

```
1. 문제는 무엇인가?
급식 시간에 줄이 너무 길어서 식사 시간이 부족하다.

2. 구체적으로 설명하면?
12시 10분쯤 급식실에 가면 줄이 꽤 길다. 한 15분은 기다리는 것 같다.
월요일이 특히 심하다.

3. 누구에게 문제인가?
1학년 학생들. 특히 4교시가 늦게 끝나는 반.

4. 왜 지금 풀어야 하는가?
밥을 빨리 먹어야 해서 소화가 안 되고, 오후에 졸리다.

5. AI는 이 문제에 대해 뭐라고 하나?
AI는 시차 급식, 배식구 추가, 모바일 주문 등을 제안했다.

6. AI 답변에서 빠진 것 또는 틀린 것은?
모바일 주문은 우리 학교에서 폰 사용 금지라 안 된다.

7. 내가 이 문제를 고른 이유는?
매일 겪는 문제라서 해결하고 싶다.
```

#### 전체 예시: 보통 (2점)

```
1. 문제는 무엇인가?
급식 줄이 길다.

2. 구체적으로 설명하면?
줄이 길어서 오래 기다린다.

3. 누구에게 문제인가?
학생들.

4. 왜 지금 풀어야 하는가?
불편하니까.

5. AI는 이 문제에 대해 뭐라고 하나?
여러 가지 해결책을 제안했다.

6. AI 답변에서 빠진 것 또는 틀린 것은?
잘 모르겠다. 다 맞는 것 같다.

7. 내가 이 문제를 고른 이유는?
급식은 매일 먹으니까.
```

#### 전체 예시: 미달 (1점)

```
1. 문제는 무엇인가?
급식

2. 구체적으로 설명하면?
(비어있음)
```

#### 단일 일지 예시: 탁월 (4점)

```
날짜/차시: 4월 8일 / 6차시

AI에게 뭘 물었나?
"Raspberry Pi Pico W에서 DHT22 온습도 센서로 측정한 값을 0.96인치 SH1106 
OLED 디스플레이에 실시간으로 표시하는 MicroPython 코드를 작성해줘. 
핀 배치: DHT22 데이터핀은 GP16, OLED는 I2C(SDA=GP0, SCL=GP1). 
2초 간격으로 읽고, 온도와 습도를 각각 한 줄씩 표시해줘."

AI가 뭘 줬나?
약 45줄의 전체 코드. dht 라이브러리와 ssd1306 라이브러리를 import하고, 
Pin 설정, I2C 초기화, while 루프에서 2초마다 센서 읽기 + OLED 출력. 
주석도 포함되어 있었음.

내가 뭘 바꿨나?
1) AI가 ssd1306 라이브러리를 import했는데, 우리 키트의 OLED는 SH1106이라 
   sh1106 라이브러리로 변경. 초기화 코드도 `SSD1306_I2C` → `SH1106_I2C`로 수정.
2) AI 코드에 에러 처리가 없어서 try-except 추가. 센서 읽기 실패 시 
   OLED에 "읽기 실패"를 표시하고, 5초 후 재시도하도록 함.
3) 온도 표시 형식을 "Temp: 25" → "온도: 25.0°C"로 변경 (한글+단위+소수점).

AI 답변의 문제점?
- SSD1306과 SH1106은 호환되지 않는 다른 드라이버 칩인데, AI는 "OLED"라고만 
  인식하고 기본값인 SSD1306을 사용함. 우리가 어떤 OLED를 쓰는지 모르기 때문.
- 센서 에러 처리가 전혀 없어서, DHT22가 간헐적으로 읽기 실패할 때 
  프로그램이 멈춤. 실제 환경에서는 반드시 에러 처리가 필요한데 AI가 누락.
- I2C 주소를 0x3C로 하드코딩했는데, SH1106은 0x3C가 맞지만 
  확인 없이 가정한 것이므로 `i2c.scan()`으로 확인하는 코드를 추가하는 게 안전함.

내가 결정한 것은?
온도가 28도 이상이면 OLED에 "환기 필요!" 경고 메시지를 추가하기로 결정.
AI는 단순 표시 코드만 제안했지만, 우리 프로젝트의 목적이 "교실 환경 
모니터링"이므로 경고 기능이 있어야 의미가 있다고 판단. 28도 기준은 
교실 냉방 가동 기준이 28도라는 학교 규정을 참고함.
```

## Part 3: AI 평가 시스템 설계 요구사항

### 핵심 기능

#### 기능 1: 개별 산출물 평가
- 입력: 학생이 작성한 텍스트 (문제 정의서 1건, AI 활용 일지 1건 등)
- 출력: 항목별 점수(1-4) + 총점 + 구체적 피드백
- 피드백 구조: "잘한 점 → 개선 제안 → 구체적 행동 가이드"

#### 기능 2: 시간에 따른 성장 분석
- 입력: 같은 학생의 산출물 2건 이상 (시간 순서)
- 출력: 항목별 변화 추이 + 성장 서술 + 시각화
- 핵심: AI 활용 일지 초반 vs 후반 비교가 가장 중요

#### 기능 3: 포트폴리오 종합 평가
- 입력: 학기 전체의 모든 산출물
- 출력: FACT 4요소별 종합 평가 + 강점/성장영역 + 생기부 서술 초안

#### 기능 4: 학급 전체 분석 (선택)
- 입력: 학급 전체의 산출물
- 출력: 학생별 역량 맵, 전체 패턴 분석, 수업 개선 시사점

### 피드백 생성 원칙

1. **칭찬 먼저**: 가장 잘한 항목을 구체적으로 먼저 언급
2. **성장 관점**: "틀렸다"가 아니라 "이렇게 하면 더 깊어질 수 있다"
3. **구체적 제안**: "더 구체적으로" (X) → "예: '몇 명이 영향을 받는지' 숫자를 넣어보세요" (O)
4. **학생 언어**: 학술 용어 대신 고등학생이 이해할 수 있는 말
5. **AI 투명성**: 이 피드백이 AI에 의해 생성되었음을 명시

### 피드백 예시

#### 문제 정의서 — 우수(3점) 학생에게:

```
[잘한 점]
문제를 구체적으로 적었어요. 특히 "12시 10분쯤"이라는 시간과 "15분 대기"라는 
수치가 좋았어요. 모바일 주문이 학교 규정상 안 된다는 점을 지적한 것도 
훌륭한 비판적 사고예요.

[더 깊어지려면]
"약 15분"을 더 정확하게 만들어보면 어때요? 실제로 내일 점심시간에 
스톱워치로 재보세요. "12:12에 줄 서서 12:27에 받았다 = 15분"처럼 
실제 데이터가 있으면 문제 정의가 훨씬 강력해져요.

또, "누구에게 문제인가"에서 "1학년 학생들"이라고 했는데, 몇 명인지, 
특히 어떤 반이 가장 영향을 받는지까지 적으면 프로젝트 범위가 명확해져요.
```

#### AI 활용 일지 — 보통(2점) 학생에게:

```
[잘한 점]
AI를 사용하고 일지를 작성한 것 자체가 좋은 시작이에요. 
"조금 바꿈"이라고 적은 건, 적어도 AI 코드를 그대로 쓰지 않고 
수정을 시도했다는 뜻이니까요.

[더 깊어지려면]
"조금 바꿈" 대신 "뭘 바꿨는지"를 적어보세요. 예를 들어 
"핀 번호를 GP15에서 GP16으로 바꿈" 이렇게요. 
그리고 "왜 바꿨는지"도요 — "우리 키트에서는 GP16에 연결되어 있어서".
이렇게 적으면 나중에 다시 볼 때도 도움이 되고, 
선생님도 여러분이 뭘 이해하고 있는지 알 수 있어요.
```

### 기술 구현 방향

#### 방향 A: Streamlit 웹앱 (추천)
- 교사가 학생 산출물을 입력(텍스트 또는 사진→OCR)
- Claude API로 루브릭 기반 평가 + 피드백 생성
- 학급 전체 대시보드, 성장 추이 차트
- 장점: 유연, 확장 가능, 교사 친화적 UI

#### 방향 B: Claude Artifact (React)
- 대화 내에서 바로 사용할 수 있는 인터랙티브 도구
- 학생 산출물 붙여넣기 → 즉시 분석
- 장점: 설치 불필요, 빠른 프로토타입

#### 방향 C: Google Sheets + Apps Script
- 스프레드시트 기반, 교사가 가장 익숙한 환경
- 장점: 기존 업무 흐름에 통합 용이

- JSON 데이터 구조

이 문서를 읽고, **Streamlit 웹앱** (또는 원하는 방식)으로 다음 기능을 구현해주세요:

1. **개별 산출물 평가**: 텍스트 입력 → 루브릭 기반 점수(1-4) + 피드백

4. **생기부 서술 초안**: 평가 결과 기반 생기부 문장 자동 생성
